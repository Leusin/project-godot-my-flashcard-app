# 코딩 컨벤션 (설계 원칙)

## 1. 책임 경계 - 자기 것만 안다

각 클래스는 자기 책임 안의 것만 참조한다.

## 2. 의존 방향 - 하위는 상위를 부르지 않는다

카드·파서·입력(하위)은 세션 규칙·저장 관리(상위)를 호출하지 않는다. 상위가 하위를 구독한다.

예시
- ✅ `StudySession`(세션 관리자)이 카드 뷰의 스와이프 이벤트를 구독해 Again/Good을 처리
- ❌ 카드 뷰가 `StudySession.OnAgain()`을 호출 ← 뷰가 세션 규칙을 배달

**판별법**: "이 객체가 없어도 상대가 존재할 수 있나?" 파서는 세션 없이도 존재해야 한다. 상대를 널가드(`if (x != null)`)로 버티고 있다면 방향이 틀렸다는 신호.

## 3. 사실 vs 해석 분리

도메인 사실과 앱의 해석을 나눈다.

예시
- 사실: "카드가 오른쪽으로 스와이프됐다", "md에서 카드 12장이 파싱됐다"
- 해석: "Good 처리", "WrongCount 증가", "세션 완료"

하위는 사실(상태·이벤트)만 노출하고, 그 의미는 세션 관리자 등 상위가 해석한다.

## 4. 뷰는 읽기만 - 로직을 구동하지 않는다

카드 UI·덱 목록·진행 표시는 상태를 **읽어 표시만** 한다. 앱 로직을 부르거나 상태를 바꾸지 않는다. 뷰가 여러 관리자의 상태를 읽는 건 최소화한다.

## 5. 순수 로직은 Godot에서 분리

Markdown 파서, 세션 큐, 진행도 직렬화 같은 순수 로직은 `Node`를 상속하지 않는 일반 C# 클래스로 만든다. Godot 씬 없이 단위 테스트가 가능해야 한다.

## 6. 주석

- 책임 주석: 각 클래스 맨 위 주석에 무엇을 하고, 무엇을 안 하는지를 한 줄로 밝힌다
- 본문 주석은 '왜'만: 코드로 드러나지 않는 제약·의도·근거만 적는다. 다음 줄이 무엇을 하는지 옮겨 적지 않는다.
- 불일치는 즉시 정정: 주석과 코드가 어긋난 채 두지 않는다. 어느 쪽이 정답인지 확인해 맞춘다 

## 7. C# 스타일

- Godot 공식 C# 컨벤션을 따른다: 클래스·메서드·프로퍼티 `PascalCase`, 지역 변수·매개변수 `camelCase`, private 필드 `_camelCase`.
- 노드 경로 문자열은 흩뿌리지 말고 `[Export]` 또는 `GetNode` 캐싱으로 한곳에 모은다.
