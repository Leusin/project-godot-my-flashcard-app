## 2026-07-22 카드 상태 라벨 + WrongCount 수동 수정 — v0.3 마지막

- **상태(NEW/LEARNING/MASTERED)는 진행도에 함께 산다.** WrongCount와 성격이 같은 카드별 앱 메타데이터라(원본 md 아님), 같은 파일에 둔다. 마크다운은 안 건드림(DESIGN이 md 태그를 향후 확장으로 미룸).
- **진행도 JSON 형식 변경 + 하위호환.** `{"질문": 3}` → `{"질문": {"wrong": 3, "status": "LEARNING"}}`. FromJson이 값 종류(숫자/객체)로 옛/새를 구분해 둘 다 읽는다 — 기존 틀린 횟수를 잃지 않는다. 옛 형식 읽기는 SceneSmokeTest가 flat `{"q":5}`를 써서 이미 검증.
- **`Progress` 이름은 유지**(책임은 넓어짐: 틀린 횟수 + 상태). 정확히는 "카드별 상태"지만 rename이 DeckStorage·App·테스트로 번져서, 실익 대비 미룸. 책임 주석만 갱신. — 냄새로 기록.
- 상태는 **Study 카드 상단에 표시**(DESIGN 위치, Tally 옆), **Editor에서 편집**(OptionButton). WrongCount는 Editor의 SpinBox로 수동 수정. 둘 다 카드 목록엔 없음(DESIGN 따름).
- **저장은 실제 변경 때만.** 내용(질문·답) 바뀌면 md 재작성, 메타(횟수·상태)만 바뀌면 진행도만 저장, 아무 것도 안 바뀌면 파일 안 건드림. 옛 값을 App이 들고 비교.

## 2026-07-22 덱 폴더 설정 (desktop) — 클라우드는 앱이 모른다

- **결정: 앱은 클라우드를 모른다 (해석 B).** "드라이브로 기기 간 동기화"를 앱에 Drive API로 넣는 건 DESIGN 비목표(클라우드 동기화·계정)라 접었다. 대신 **덱 폴더만 지정 가능**하게 하고, 그 폴더를 Drive 동기화 폴더로 두면 동기화는 OS가 한다. 앱은 로컬 파일만 안다.
- **덱 폴더만 이동, 진행도·설정은 `user://` 고정.** 진행도는 파생 데이터고 파일 이름으로 키를 잡아, 덱이 어디 있든 짝이 맞는다. `DeckStorage`에 정적 `_decksDir` + `SetDecksDir`. 절대 경로(예: `C:/.../Flashcards`)도 그대로 받는다 — desktop Godot의 FileAccess/DirAccess가 절대 경로를 처리.
- **폴더 바꾸면 그냥 전환**(자동 복사 안 함). 그 폴더가 곧 원본이라 앱이 파일을 옮기지 않는다. 같은 이유로 예시 덱은 **기본 폴더가 비었을 때만** 넣는다 — 사용자 폴더(동기화 폴더 등)엔 안 끼워넣는다.
- 설정을 App이 통째로 들고 read-modify-write. 예전엔 `SaveSettings(new AppSettings{ LastDeckFile })`로 덮어썼는데, 이제 필드가 둘이라 그러면 `DeckDir`이 날아간다. `_settings` 필드로 보관.
- UI: 덱 목록에 현재 폴더 경로(OS 절대 경로로 표시) + "폴더 변경"(네이티브 폴더 선택기).
- **Android는 별개 난관**(SAF 필요)이라 v1.0 리스크에 기록. Android 기본은 `user://decks`.
- 테스트 119 → 126.

## 2026-07-22 v0.3 카드 추가/삭제 — 편집 CRUD 완성

- **추가**: 카드 목록 `＋ 카드 추가` → 빈 편집기. `_editingIndex = -1`이 "기존 카드 아님" 신호 → 저장 때 덱 끝에 붙인다. 새 카드는 옛 질문이 ""이라 `Rename`이 no-op(진행도 안 건드림).
- **삭제**: 편집기 `삭제` 버튼(`DeleteRequested`, 자동 저장과 별개 경로). App이 그 자리 카드를 빼고 md 재저장. **남은 카드가 그 질문을 안 쓸 때만** 진행도 `Remove` — 중복 질문의 기록을 실수로 지우지 않으려는 확인.
- 새 카드에서 `삭제`는 지울 게 없어 그냥 목록으로(= 취소). "삭제=이 카드 버리기"로 기존/신규 모두 자연스럽게 한 버튼.
- 순수 로직에 `Progress.Remove` 추가(`Rename`의 형제). 편집의 두 부품(`Rename`·`Remove`) + `DeckWriter`로 CRUD가 다 섰다.
- 화면은 안 늘었다(편집기·목록에 버튼만). 이번에도 App만 해석 추가.

## 2026-07-22 v0.3 Card Editor — "앱 수정 → 마크다운 반영" 실증

- **v0.3 핵심이 실제로 돌았다.** 카드 탭 → 편집기 → 나가면 md 재저장 + 진행도가 카드를 따라온다. `DeckWriter.ToMarkdown`과 `Progress.Rename`이 여기서 처음 엮인다.
- **저장은 ←로 나갈 때 자동**(사용자 선택, 모델 A의 "끝날 때 저장"과 일치). 편집기 출구가 ← 하나뿐이라 별도 저장 버튼·취소 없음. 편집기는 "저장하라"를 모르고 입력값(질문,답)만 `EditingDone`으로 올린다 — 저장 해석은 App.
- **카드 정체성은 위치(index)로 든다.** 질문이 편집으로 바뀌니 질문을 키로 쓸 수 없다. App이 편집 시작 시 `_editingIndex`·옛 질문/답을 보관 → 저장 때 그 자리 카드를 교체하고 `Rename(옛 질문, 새 질문)`.
- **안 바뀌면 파일을 안 쓴다.** 자동 저장이라 카드를 열었다 그냥 나오는 일이 잦은데, 매번 md를 다시 쓰면 손으로 맞춘 서식이 정규화돼 버린다. 옛 질문/답과 비교해 실제 변경 때만 기록. **단, 한 번이라도 저장하면 덱 전체가 우리 writer의 정규 형식으로 재작성된다** — 앱이 파일 주인이라 수용(Import 후 앱 저장소가 원본).
- **빈 질문은 무시.** 질문을 비우고 나가면 파서가 그 카드를 버려 삭제가 돼버리므로, 저장하지 않고 카드를 그대로 둔다 (삭제는 별도 기능).
- **질문 입력은 LineEdit(한 줄).** 마크다운에서 질문은 `# 한 줄`이라 여러 줄이면 왕복이 깨진다 — UI에서 원천 차단. 답은 TextEdit(여러 줄).
- 화면 넷 → 다섯. 이번에도 App 말고 새로 고친 화면 로직 없음(카드 목록 행을 Label→Button으로 바꿔 `CardChosen`만 추가). `ShowOnly(screen)` 헬퍼로 정리 — 화면 추가 시 여기 한 줄만 는다.
- 테스트 105 → 111.

## 2026-07-22 v0.3 순수 로직 + Card List

- **편집의 두 부품을 순수 로직으로 먼저 깔았다** (편집기 전에, 테스트로 검증):
  - `Progress.Rename(옛→새)` — 질문을 고쳐도 진행도가 카드를 따라간다. 겹치면 합치기(merge, 데이터 안 잃음).
  - `DeckWriter.ToMarkdown` — `DeckParser.Parse`의 짝. 기준은 왕복 안정 `Parse(ToMarkdown(cards)) == cards`. 빈 답은 줄을 안 만든다. **한계**: 답에 `# ` 시작 줄이 있으면 왕복이 깨진다(읽기 쪽도 원래 같은 한계, 지금은 안 막음).
- **Card List 화면 (읽기 전용).** Deck 상세 화면을 v0.2에서 건너뛰어서, 진입은 DESIGN Study 상단바의 **✏**로 잡았다. Study `EditRequested` → App이 카드 목록 표시, 카드 목록 `←` → Study 복귀. App이 `_currentDeck`을 들고 Study·카드 목록이 같은 덱을 보게 한다.
- 화면이 셋 → 넷이 됐는데 App 말고 고친 곳은 없다(각 화면은 자기 시그널만). "화면 추가 시 App만 고친다" 규칙이 실제로 지켜짐.
- 카드 탭 → 편집은 아직 없음(읽기 전용이라 행은 Label). 편집기 붙일 때 행을 버튼으로 바꾸고 `CardChosen` 배선.
- 테스트 100 → 105.

- **스펙 뒤집음.** 기존 DESIGN: "Question이 변경되면 진행도 초기화." 바뀐 결정: **질문을 고쳐도 진행도가 카드를 따라온다.**
- 근본 원인은 진행도 키가 Question이라는 것. 후보 둘을 놓고 골랐다:
  - **모델 A (채택)** — 카드가 편집 세션 동안만 런타임 정체성을 갖고, 화면을 나갈 때 진행도 키를 `옛 질문 → 새 질문`으로 옮긴다. 진행도 JSON은 계속 Question을 키로 저장해 사람이 읽을 수 있다. **마크다운은 안 건드린다.**
  - 모델 B (기각) — 마크다운에 카드마다 영구 ID 토큰을 심는다. 앱 밖 수정까지 커버되지만 "최대한 단순한 마크다운" 원칙(DESIGN 비목표)을 내줘야 한다.
- **왜 A**: 이 프로젝트의 핵심 가치가 "PC에서 md 하나만 편집". 그 원칙을 지키면서 인앱 편집 케이스는 전부 해결된다. B가 더 커버하는 "PC에서 직접 질문 수정 시 진행도 유지"는 원칙을 내주는 거래라 접었다 — **의도적 한계로 수용.**
- 미결(구현 시 정함): 이사한 키가 같은 덱의 다른 질문과 겹칠 때 합치기/덮어쓰기. 순수 로직이라 진행도 이사 함수부터 테스트로 만들 자리다.

## 2026-07-22 v0.2 여러 덱 — 저장소와 화면 전환

- **원본은 앱 저장소 하나로 통일.** `user://decks/`에 있는 md만 덱으로 본다. `res://sample_deck.md`는 덱이 하나도 없을 때만 복사되는 예시로 물러났다(사용자가 지운 덱을 되살리지 않도록 "비었을 때만"). 앱이 `res://`를 직접 읽는 경로는 없앴다.
- **진행도는 덱마다 따로 파일.** 카드 식별자가 Question 텍스트라, 덱이 늘면 다른 덱의 같은 질문끼리 진행도를 공유해버린다. `user://progress/<덱이름>.json`으로 나눴다. **기존 `user://progress.json`은 이제 아무도 읽지 않는다** — 옮기지 않고 버린다(샘플 덱 WrongCount뿐이라 이전 코드를 영구히 남길 이유가 없다).
- **진입점이 `study.tscn` → `app.tscn`으로 바뀌었다.** `App`이 화면 전환(덱 목록 ↔ Study)과 마지막 사용 덱 기억만 맡는다. `Study`는 덱을 스스로 고르지 않고 `StartDeck(deckFile)`로 받으며, 나가고 싶다는 **사실**(`ExitRequested`)만 위로 알린다 — 어디로 갈지는 `App`의 해석.
- 실행 흐름: 마지막 덱이 있으면 바로 Study, 없으면 덱 목록. DESIGN의 Deck 상세 화면은 v0.3 Card List와 함께 보기로 하고 이번엔 넣지 않았다.
- **`DeckStorage`는 static.** 경로 계산과 파일 IO뿐이라 상태가 없다. 인스턴스로 두면 App·Study 중 누가 소유하냐는 질문만 생긴다.

- 가져온 덱은 바로 열지 않고 목록에 넣는다 — 파일을 고른 것과 공부를 시작하는 것은 다른 결정.
- 테스트 56 → 86개. 앱 저장소를 건드리는 테스트는 전용 덱(`__test_deck.md`)만 쓰고 지우며, `settings.json`은 백업·복원한다.
- **`src/screens/<화면>/`으로 폴더를 나눴다.** 화면이 셋이 되며 `src/` 바로 아래가 평평해졌다. 앱(진입점·저장소)은 `src/`에, 화면은 씬과 스크립트를 한 벌로 묶는다 — CODE_GUIDE의 화면 흐름 그림과 폴더 모양이 같아진다.
  - 에디터 밖에서 옮기면 `.tscn`의 `path="res://..."`를 손으로 전부 고쳐야 한다. `.uid`는 스크립트와 함께 따라가므로 짝지어 옮기면 된다.
  - **옮긴 직후 첫 테스트 실행은 실패한다** — Godot이 프로젝트를 다시 스캔하는 중이라 옛 경로로 읽는다. 한 번 더 실행하면 통과한다. 이걸 모르면 경로를 잘못 고친 줄 알고 헤맨다.

## 2026-07-22 카드 화면·세션 규칙 확정

- **한 세션은 덱을 한 바퀴 돈다.** Again을 눌러도 그 카드는 이번 세션에 다시 나오지 않는다. 그 결과 `StudySession`이 판정을 구분할 이유가 없어져 `Next()` 하나로 합쳤다. 큐는 판정 이름을 모르고 WrongCount 증가는 `Study`의 몫.
- 카드 UI 확정: 탭으로 앞뒤 토글, 뒷면에서 질문 축소(폰트·색·3줄 제한 + `…`), 질문:답 세로 분할, 정렬(질문 가운데 / 답 왼쪽·위), 판정 버튼 상시 표시(`← Again` / `Got it →`).
- WrongCount는 카드 좌측 상단에 작대기 그림. **`_Draw()`로 직접 긋는다** — 유니코드 작대기는 폰트 지원이 들쭉날쭉하고 5개 묶음의 사선을 문자로 표현할 수 없다. `TallyMarks`는 받은 수를 그리기만 하고 무엇을 세는지는 모른다.
- **레이아웃 수치의 출처는 `CardView` 상수. 씬에는 두지 않는다.** 값이 두 곳에 흩어지지 않고, 에디터가 씬을 덮어써도 살아남는다. 대신 에디터 미리보기에는 반영되지 않고 라벨 경고도 남는다 — 정상이다.
- 컨벤션 추가: 인스턴스 멤버 접근에 `this.` 강제. `this.`가 없다는 것이 곧 "인스턴스 상태가 아니다"라는 신호가 된다.
- PowerShell 래퍼(`run_tests.ps1`·`apply_settings.ps1`) 제거. 한 줄짜리 godot 명령을 감쌀 뿐이라 AGENT.md에 명령만 남겼다.
- **안드로이드 조사 (설치 보류)**: 익스포트 템플릿·JDK 17·Android SDK·adb 전부 미설치, 합계 2~3GB. ROADMAP v1.0에 준비물 체크리스트로 옮겼다. 실기기는 있고 USB 연결 가능. **최대 미지수는 C# 안드로이드 빌드에 .NET 워크로드가 필요한지 여부.**
- 테스트 29 → 56개.

## 2026-07-21 에디터가 파일을 덮어쓰는 이유 (원인 규명) ★

하루 종일 시달린 두 현상의 원인을 에디터 설정에서 찾았다.

- **`run/auto_save/save_before_running` (기본 true)** — F5/F6로 실행할 때마다 열린 씬을 전부 저장한다. 탭이 오래된 내용을 들고 있으면 실행하는 순간 그게 덮어쓰기가 된다. 리팩터한 `StudyView.cs`가 통째로 되돌아가 앱이 깨진 적이 있다.
- **`run/window_placement/game_embed_mode` (기본 0)** — Godot 4.4+는 게임을 에디터 안 Game 탭에 끼워 실행한다. 창 크기가 프로젝트 설정 대신 패널 크기를 따르므로 레이아웃이 다르게 보인다.
- 대응: AI가 파일을 고치는 동안에는 에디터를 닫고, 작업 후 테스트로 확인한다. 창 크기·비율 판정은 에디터 밖에서 실행해서 한다.

## 2026-07-21 화면과 레이아웃

- **화면은 해상도를 다 쓰고(`stretch/aspect = expand`), 카드 비율은 `AspectRatioContainer`가 지킨다.** 측정으로 확인: 720×1280 / 1200×700 / 500×1400 세 화면 모두 카드 비율 0.700.
- 긴 질문·답은 `ScrollContainer`(세로 Auto)로 감싼다 — 내용이 넘칠 때만 스크롤바가 생긴다. 가로 스크롤은 꺼야 `autowrap`이 동작한다.
- 실행 창 크기(`window_*_override`)는 설정 스크립트가 관리하지 않는다. 테스트하며 자주 바꾸는 값이라 스크립트가 되돌려놓으면 방해가 된다.
- **[경계] 설정 스크립트는 "선언한 값을 넣기"만 한다.** 선언에서 빼도 `project.godot`에 남은 값은 지워지지 않으니, 뺄 때는 두 곳 다 확인할 것.
- **[교훈] 헤드리스에서는 마우스 클릭 라우팅이 동작하지 않는다.** 대조군으로 놓은 순수 `Button`도 눌리지 않았다. 클릭·탭 동작은 사람이 눈으로 확인해야 한다.
